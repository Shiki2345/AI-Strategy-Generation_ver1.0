### A1. `supervise/get_sensitive_url_config`

**来源：D1. 公司监控配置管理 / D9. 敏感 URL 管控**

| 路由 | Handler | 权限 | 说明 |
|------|---------|------|------|
| `POST supervise/get_sensitive_url_config` | `api_get_sensitive_url_config` | 管理员 | 获取 `is_block_sensitive` + 敏感 URL 列表 |

**关联数据模型：**

`CompanySupervise`（`t_company_supervise`）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `is_block_sensitive` | int=1 | 是否阻断敏感 URL |

`SuperviseSensitiveUrlCompanyConfig`（`t_supervise_sensitive_url_company_config`）：

| 字段 | 说明 |
|------|------|
| `sensitive_url_id` | 关联系统预设敏感 URL ID（`t_access_rule_url`） |
| `is_active` | 该公司是否启用此敏感 URL |

**注意**：`CompanySupervise.is_block_sensitive` 是全局总开关，为 `0` 时所有敏感 URL 均不生效。

**时序图（浏览器获取敏感 URL 配置）：**

```
浏览器客户端         api.py              CompanySupervise    SuperviseSensitiveUrlCompanyConfig
     │                  │                     │                        │
     │ POST account/    │                     │                        │
     │ config           │                     │                        │
     │ {account_id}     │                     │                        │
     │─────────────────►│                     │                        │
     │                  │ from_company_id()   │                        │
     │                  │────────────────────►│                        │
     │                  │◄── {is_block_       │                        │
     │                  │     sensitive: 1}   │                        │
     │                  │                     │                        │
     │                  │ if is_block_sensitive:                        │
     │                  │   SELECT sensitive_urls                       │
     │                  │   JOIN t_access_rule_url                      │
     │                  │   JOIN company_config                         │
     │                  │─────────────────────────────────────────────►│
     │                  │◄── [{url, is_active}] ──────────────────────│
     │◄── {warning_rules│                     │                        │
     │  sensitive_urls} │                     │                        │
```

**关联数据模型 `CompanySupervise`（`t_company_supervise`）：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `company_id` | int | 公司 ID |
| `supervise_day` | int=1 | 最长可查看的日志天数 |
| `supervise_user_num` | int=0 | 用户模式额度上限 |
| `supervise_account_num` | int=0 | 账号模式额度上限 |
| `first_supervise_time` | int=0 | 首次开启监控时间戳 |
| `max_view_times` | int=1 | 日志最大查看次数 |
| `is_show_tips` | int=1 | 是否展示监控提示 |
| `supervise_type` | int=0 | 0=账号模式, 1=用户模式 |
| `is_block_sensitive` | int=1 | 是否阻断敏感 URL |
| `is_supervise_enabled` | int=0 | 是否曾经开启过监控 |
| `tryout_start_time` | int=0 | 试用开始时间 |
| `tryout_end_time` | int=0 | 试用结束时间 |
| `tryout_status` | IntEnum | PENDING=1 / IN_PROGRESS=2 / EXPIRED=3 |
| `tryout_count` | int=0 | 试用次数 |

**关键枚举：**

```python
class SuperviseType(IntEnum):
    ACCT = 0   # 以账号为监控单位（关注"用哪个账号"）
    USER = 1   # 以用户为监控单位（关注"谁在操作"）

class SuperviseTryoutStatus(IntEnum):
    PENDING    = 1   # 未开始试用
    IN_PROGRESS = 2  # 试用中
    EXPIRED    = 3   # 试用已过期
```

**关键业务规则：**

| 规则 | 说明 |
|------|------|
| **监控模式保持** | 支付回调仍按 `CompanySupervise.supervise_type` 读取模式，再写入新额度，避免模式被重置 |
| **切换模式联动** | 切换 `supervise_type` 时同步互换 Detail 的 `user_num/account_num`，并重置所有预警计划目标为 ALL |

### C6. `supervise/account/config`（核心接口）

**来源：D2. 账号监控管理 / D6. 预警计划管理**

| 路由 | Handler | 权限 | 说明 |
|------|---------|------|------|
| `POST supervise/account/config` | `api_supervise_account_config` | 客户端 | 会话开始时获取监控配置和预警规则（详见 D6 预警计划管理） |

**业务说明：**

此接口是浏览器客户端与监控系统的核心数据同步点。会话开始时下发**全量预警规则**给客户端，客户端根据 `event_type` 自行决定走哪条路径：

- `event_type` 为 `AMAZON_`* → 等用户触发对应 Amazon 操作时，调 `preset/check` 由后端计算（路径 A）
- `event_type` 为 `COMMON_OPERATION` / `CUSTOM` → 本地直接匹配判断（路径 B）

**预警触发的两条路径（共用同一套数据源 `t_supervise_warning_plan` + `t_supervise_warning_rule`）：**

| 路径 | 规则类型（event_type） | 判断方 | 触发方式 | 数据来源 |
|------|----------------------|--------|---------|---------|
| **路径 A** | `AMAZON`_*（折扣/标题/图片/价格） | **后端** | 客户端在用户提交后调用 `preset/check` 接口来获取是否需要告警和拦截操作 | `t_supervise_warning_plan` + `t_supervise_warning_rule` |
| **路径 B** | `COMMON_OPERATION` / `CUSTOM`（访问/复制/下载/DOM点击） | **客户端本地** | 会话开始时从 `account/config` 下载全量规则，本地实时判断 | `t_supervise_warning_plan` + `t_supervise_warning_rule` |

**路径 B 的完整流程：**

```
① 会话开始
   客户端 POST supervise/account/config
        → 后端调 get_warning_config()
          查 t_supervise_warning_rule JOIN t_supervise_warning_plan（匹配当前账号/用户）
        → 返回 websites[]（含 url_pattern、re_pattern、warning_events[{event_type, config, ...}]）
        → 客户端缓存到本地

② 会话运行中（本地实时判断，仅针对 COMMON_OPERATION / CUSTOM 规则）
   用户触发操作（访问网页 / 复制 / 下载 / 点击 DOM 元素）
        → 客户端匹配 url_pattern / re_pattern
        → 比对 config（如 access=true / clipboard=true / click_dom=[domId] 等）
        → 命中 → extra.warning.result = true（+ rule_ids / event_ids / context_id）

③ 打包上传
   行为事件连同 warning 判断结果一起打包进 zip
        → OSS 直传 → FC 解压 → is_warning=1 写入 SLS
        → warning_event_handler 异步写 t_log_supervise_warning_event + 更新 warning_count
```

### E18. `supervise/warning_plan/preset/check`（核心接口）

**来源：D6. 预警计划管理 / D7. Amazon SKU 保护**

| 路由 | Handler | 说明 |
|------|---------|------|
| `POST supervise/warning_plan/preset/check` | `warning_plan_check` | 浏览器实时预警检测（路径 A，Amazon 三项，后端判断） |

**Amazon 三项规则引擎：**

每种 Amazon 场景对应独立的 `RuleConfig` 子类，均实现 `check(value) → (is_warning, is_stop_user)`：

| 规则类 | 触发条件 | `sort_value`（规则排序键） |
|--------|---------|----------------------|
| `AmazonPercentigeOffConfig` | `discount > condition.discount`（默认20%） | `condition.discount` |
| `AmazonCouponDiscountConfig` | `discount > condition.discount`（默认20%） | `condition.discount` |
| `AmazonTitleEditConfig` | `count >= max_count`（默认2次/1天） | `max_count / days` |
| `AmazonImageEditConfig` | `count >= max_count`（默认3次/7天） | `max_count / days` |
| `AmazonLowPriceConfig` | `percent < condition.percent`（默认90%） | `-condition.percent` |
| `WarningConfig`（通用） | 勾选的操作类型发生时 | — |

`StopUser` 字段控制是否在触发后阻断用户操作（`is_stop_user=True`）。

**目标范围解析逻辑：**

```
get_warning_plan(company_id, account_id, user_id, is_boss)
  ├── 查找 target_account=ALL 的计划
  ├── 查找 account_id 关联的 ACCOUNT 计划
  ├── 查找 target_user=ALL 的计划
  ├── 查找 target_user=EXCEPTBOSS（非Boss用户才匹配）
  ├── 查找 user_id 关联的 USER 计划
  ├── [非Boss] 查找用户所属部门关联的 DEPARTMENT 计划
  └── [非Boss] 查找用户所属角色关联的 ROLE 计划
```

**申请通过（Pass Request）机制：**

```
get_user_request_pass_rule()
  查询 t_access_rule_request
    WHERE request_type=2     -- 申请类型：预警豁免
    AND status=2             -- 状态：已批准
    AND 在有效期窗口内
  → 返回该用户已被豁免的 rule_ids 集合

check_have_request_waiting_pass()
  查询 status=1（等待审批中）
  → 返回 is_have_request=True 给浏览器
```

**时序图（浏览器实时预警检测）：**

```
浏览器客户端        warning_plan_api    WarningPlanService    SuperviseWarningRule    通知系统
     │                   │                    │                     │                   │
     │ POST preset/check │                    │                     │                   │
     │ {store_id,        │                    │                     │                   │
```

**价格计算规则：**

```
主套餐价格 = package.real_price × count
           − 优惠券抵扣（仅作用于主套餐部分）

监控加量包价格 = supervise_package.real_price × count × diff_days / period
              × company.supervise_quota_discount（公司专属加购折扣）

最终支付 = 最终主套餐价 + 最终加量包价
```

`diff_days` 取值规则：
- **新购 / 续费**：同主套餐的 `period`（天数）
- **单独加购**：取 `company_member.expiry_time` 剩余天数（按天折算，不足整天按整天）

**关键业务规则：**

| 规则 | 说明 |
|------|------|
| **防重入锁** | `vip_member_order:{company_id}` Redis key，INCR > 1 时拒绝，TTL 10 分钟 |
| **金额前后端双校验** | 后端重新计算，与前端 `pay_price.paid` 精确到分完全一致，否则报错 |
| **额度写入策略** | 主套餐（`data_type=Member`）→ 覆盖；加量包 → 累加 |
| **续费 status=0** | 提前续费时 Detail 写 `status=0`，到期时由定时脚本 `member_expiry.py` 激活 |

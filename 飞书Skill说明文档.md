## 二、功能说明

### 2.1 核心能力

| 能力 | 说明 |
|------|------|
| **网页黑名单**（已限制模式） | 按 URL 完全屏蔽指定页面 |
| **网页行为管控**（可访问模式） | 允许打开但限制 7 项页面内操作：复制/文件上传/文件下载/打印/鼠标点击/键盘输入/查看密码框 |
| **审批控制** | 两个勾选项：仅 Boss 可审批 / 不允许申请访问 |
| **生效范围控制** | 按成员/部门/角色、按账号、按时段（周期+时间段）生效，支持反选模式 |

### 2.2 对话流程（5 Phase）

1. **Phase 1 — 意图识别 + 能力边界检查**：判断需求在平台支持范围内 / 条件支持 / 不支持
2. **Phase 2 — 多轮对话收集参数**：逐项收集生效成员、账号、时段、URL 规则、审批限制
3. **Phase 3 — 确认汇总**：校验策略名称 ≤24 字、描述 ≤400 字、域名精准度
4. **Phase 4 — 生成表单输出**：输出 JSON，字段名与平台表单标签 1:1 对应
5. **Phase 5 — 交付**：提供调整选项

### 2.3 平台能力边界

**支持**：URL 级屏蔽/放行、7 项行为控制、按人员/账号/时段/角色生效、审批控制、通配符匹配、元素级控制。

**条件支持**：数据隔离（需先问是否独立账号）。

**不支持**（会指路到平台其他模块）：
- 字段级权限 → 网页元素采集
- 多级审批链 → 仅支持 Boss 审批 / 禁止申请
- API 权限 → 开发者模式配置
- 防截图 → 屏幕水印设置
- 监控操作 → 事中监管设置
- 拦截记录 → 访问拦截日志

### 2.4 URL 语法规则

1. 所有 URL 必须以 `https://` 开头
2. `*` 通配符：`https://*.amazon.*/advertising/*` 覆盖所有亚马逊站点
3. `$` 终止符：`https://主域名/$` 仅匹配主域名屏蔽子路径
4. `?` 参数保留：控制是否精确匹配查询参数

## 三、使用方法

### 3.1 部署到飞书 Aily

1. 获取最新打包文件：项目根目录下的 `access-policy-builde_ver2.0.zip`（40 KB）
2. 登录飞书 Aily 管理后台，进入 Skill 管理页面
3. 上传 zip 文件。上传成功后系统会自动解压，识别 `SKILL.md` 为主入口、`assets/` 下为数据资产
4. 在对应的 Agent / 应用中启用此 Skill

**打包结构必须符合要求**：zip 根目录直接放 `SKILL.md` 和 `assets/` 文件夹，不要套外层包装文件夹，否则飞书会判定为无效文件。

### 3.2 触发方式

直接在飞书 Aily 对话框输入与访问策略相关的需求，Skill 会自动识别并进入配置流程。常见触发关键词：

| 关键词类别 | 示例 |
|-----------|------|
| 限制类动词 | 禁止访问、限制、不允许、屏蔽 |
| 策略对象 | 访问策略、黑名单、网页、页面 |
| Amazon 场景 | Seller Central、checkout、金融页面、广告板块 |
| 控制类型 | 审批、boss 可审批、只读、不允许下载 |

无关话题（闲聊、广告投放咨询、平台使用教程）会被直接拒绝，不消耗多余 token。

### 3.3 典型对话场景

#### 场景 A：单页面屏蔽

```
用户：禁止客服访问广告页面
AI：好的，我帮你配置。还需要确认几个问题：
    1. 生效时段是一直生效还是指定时段？
    2. 对所有平台账号生效还是指定账号？
    3. 需要"不允许申请访问"还是"仅 BOSS 可审批"？
用户：一直生效，所有账号，仅 BOSS 审批
AI：【输出 JSON 配置】
```

#### 场景 B：角色批量配置

```
用户：帮我给客服设置权限：只能看买家消息和订单，其他都屏蔽
AI：识别为客服角色策略，调用 role_templates 中的客服模板。
    【输出完整 JSON，订单/消息页用可访问+只读模式，其他 Amazon 模块全部屏蔽】
```

#### 场景 C：数据隔离（条件支持）

```
用户：我想让 6 组的 Case 别人看不到
AI：需要先确认：6 组目前使用的是独立的平台账号，还是与其他组共用？
用户：独立账号
AI：好，那可以实现。我会配置"生效成员：除 6 组外所有部门"，
    在 6 组账号上屏蔽 Case 页面。继续收集其他参数...
```

#### 场景 D：不支持的需求（会指路）

```
用户：我想让员工申请访问财务页要先主管审批再 BOSS 审批
AI：平台支持两种审批方式：「仅 BOSS 审批」和「禁止申请访问」，
    逐级审批目前还不支持。
    如果"仅 BOSS 审批"能满足您的需求，我可以帮您配置。
```

### 3.4 输出结果的使用

AI 生成 JSON 配置后，按以下步骤操作：

1. **登录紫鸟浏览器平台**
2. **导航路径**：`安全监管 > 安全拦截 > 访问策略设置 > 新增`
3. **逐字段填入**：AI 输出的 JSON 字段名（策略名称、生效成员、生效时段、访问审批限制...）与平台表单标签**完全一一对应**，无需二次翻译
4. **URL 规则**：把 `tab_指定网页生效策略` 下的每条 URL 规则和对应的操作行为控制填入「指定网页生效策略」tab
5. **全局策略**：把 `tab_所有网页通用策略` 填入「所有网页通用策略」tab
6. **保存并启用策略**

### 3.5 使用注意事项

| 事项 | 说明 |
|------|------|
| 不假设默认值 | AI 不会替你决定"一直生效""所有账号"这类参数，会主动追问。如果你想快速得到结果，可以在第一句话里把所有参数说完 |
| 字符长度限制 | 策略名称 ≤ 24 字，策略描述 ≤ 400 字。AI 会自动截断并提示 |
| URL 必须规范 | 所有 URL 以 `https://` 开头，跨站点用 `*.amazon.*` 通配符，买家号用精确域名 `https://www.amazon.com/` |
| 角色/部门名称 | AI 不知道贵公司实际的角色和部门名称，只会原样使用你说的名称。建议提前在「企业管理 > 成员管理」确认准确名称 |
| 条件支持需求 | 数据隔离类需求会先被追问账号结构，不要因为 AI 没立刻生成策略就以为它不支持 |
| 边界意识 | AI 只处理访问策略相关问题。其他平台问题（如拦截记录、屏幕水印、企业日志等）会被指路到对应模块 |

## 四、测试方法

### 4.1 测试目标

验证 LLM 能否准确将**用户中文口语描述** → **映射到正确的 URL 规则**，并输出结构正确的策略 JSON。

### 4.2 五阶段测试方案

#### 阶段 1：RAG 基线命中测试
- **方法**：从 `rag_simple_db.json` 随机抽 30 条，用 `user_description` 原样作为输入
- **示例**：输入"添加商品" → 期望输出 `https://*.amazon.*/product-search`
- **期望命中率**：≥90%
- **诊断意义**：不过关说明 RAG 未被正确加载

#### 阶段 2：别名命中测试
- **方法**：从每条记录的 `aliases` 取非首项作为输入
- **示例**：输入"上新"（rag_001 的 aliases[2]）→ 期望输出 `https://*.amazon.*/product-search`
- **期望命中率**：≥75%
- **诊断意义**：不过关说明 LLM 没有充分利用 aliases 数据

#### 阶段 3：场景化口语测试（核心指标）
- **方法**：手工构造 20 条真实用户语气的描述
- **示例**：
  - 输入"不想让客服看到进账的钱" → 期望：payments 类页面
  - 输入"禁止实习生改价格" → 期望：pricing 类页面
  - 输入"把发 FBA 的页面锁了" → 期望：fba/sendtoamazon 类页面
- **期望命中率**：≥60%
- **诊断意义**：这是真实使用场景，直接决定 demo 是否能用

#### 阶段 4：模块聚合测试
- **方法**：触发"屏蔽 XX 下所有页面"指令
- **示例**：
  - 输入"屏蔽设置下所有页面" → 期望输出 12 条 URL（全部子菜单）
  - 输入"屏蔽金融页面" → 期望输出 10 条 URL
- **期望完整率**：100%（少一条即失败）

#### 阶段 5：边界行为测试
- **歧义输入**："屏蔽报表" → 期望 LLM 追问"业务/税务/财务哪种？"
- **虚构输入**："屏蔽亚马逊 XX 页"（虚构页面）→ 期望 LLM 坦诚说未匹配到，不瞎编
- **跨站输入**："屏蔽日本站账户页" → 期望使用通配符 `*.amazon.*` 而非逐站点列

### 4.3 测试规模

| 阶段 | 样本量 | 优先级 |
|------|--------|--------|
| 阶段 1 | 30 | 必测 |
| 阶段 2 | 30 | 必测 |
| 阶段 3 | 20 | **最重要** |
| 阶段 4 | 5 | 必测 |
| 阶段 5 | 5 | 必测 |

### 4.4 评分标准

| 评分 | 定义 |
|------|------|
| ✅ 命中 | 输出的 URL 与期望完全一致，或是更合理的通配符变体 |
| ⚠️ 部分命中 | 聚合场景下遗漏部分条目（按输出对的条数/期望条数计分） |

## 六、整体判定

### 6.1 判定标准

Skill 上线可用的**最低标准**：

| 阶段 | 最低要求 | 说明 |
|------|---------|------|
| 阶段 1 | ≥85% | RAG 加载正常的基本证据 |
| 阶段 2 | ≥70% | 别名数据有效利用 |
| 阶段 3 | ≥55% | 真实使用场景可用 |
| 阶段 4 | 100% | 聚合指令必须完整，少一条即失败 |
| 阶段 5 | ≥4/5 | 边界行为合理 |

### 6.2 判定结果

- [ ] **全部通过** → Skill 可上线使用
- [ ] **部分通过** → 根据失败阶段做针对性优化（见 5.3）
- [ ] **未通过** → 排查根本原因后重新打包

### 6.3 失败诊断与修复路径

| 失败阶段 | 可能原因 | 修复方向 |
|---------|---------|---------|
| 阶段 1 低 | RAG 文件未被加载 | 检查 zip 结构、SKILL.md 中对 `assets/rag_simple_db.json` 的引用 |
| 阶段 2 低，阶段 1 正常 | LLM 只用 user_description | 在 SKILL.md 强调"优先用 aliases/keywords 匹配" |
| 阶段 3 低，阶段 2 正常 | 语义泛化不足 | 扩充 aliases 覆盖口语；或换更强模型 |
| 阶段 4 缺条目 | SKILL.md 被截断或未读完 | 检查 SKILL.md 总长度是否超出模型上下文限制 |
| 阶段 5 瞎编 | 缺少诚实边界约束 | SKILL.md 加强"匹配不到就追问，不要编造 URL" |
